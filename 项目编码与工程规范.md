# 通用 Zig 项目编码与工程规范

## 前言

本规范目标：统一代码风格、降低维护成本、规避内存问题、提升工程健壮性、保证长期可迭代性。

## 一、通用命名规范（全局强制）

适用于所有变量、函数、类型、常量、文件、模块，全项目统一。

### 1\.1 基础命名大小写规则

- **结构体/枚举/联合体/类型别名**：`PascalCase`

- **函数/变量/字段/模块文件**：`snake_case`

- **全局常量/静态配置**：`SCREAMING_SNAKE_CASE`

- **私有内部辅助函数**：下划线前缀 `_private_func`

### 1\.2 命名语义约束

- 禁止自定义非通用缩写，仅允许行业公认缩写（ctx、err、idx、buf、ptr 等）

- 命名完整表意，杜绝模糊命名：`proc`/`tmp`/`data` 无意义命名禁止使用

- 局部短作用域变量可极简（i、j、idx），长生命周期变量必须语义完整

### 1\.3 对外 API 命名规范

- 跨模块/对外暴露接口统一格式：`模块_对象_行为`

- C ABI 导出接口，全部统一前缀，杜绝随意命名

## 二、访问控制与模块可见性规范（强制最小权限）

核心原则：**默认私有，仅必要公开**，从语言层面杜绝滥用导出。

### 2\.1 符号导出规则

- 所有函数、结构体、常量默认不写 `pub`，默认模块私有

- 仅跨目录、跨模块调用的实体，才允许标记 `pub`

- 内部辅助结构体、工具函数、状态字段，一律私有，禁止导出

### 2\.2 结构体成员访问约束

- 禁止跨模块直接读写结构体字段

- 所有字段读写、状态修改，必须通过结构体成员方法完成

- 对外封装优先使用不透明指针，隐藏所有内部实现细节

### 2\.3 平台代码隔离规则

- 所有系统、平台、OS 专属逻辑，全部收敛到独立 `os/` 层

- 业务核心代码、通用工具代码**禁止出现任何平台判断逻辑**

- 条件编译仅允许在适配层使用，上层完全平台无关

## 三、函数设计通用规范（高可维护性核心）

### 3\.1 函数尺寸与职责约束

- 单函数严格遵循**单一职责原则**

- 常规函数行数控制在 **20–60 行**

- 超过 80 行的函数必须拆分为私有子辅助函数

- 禁止巨型函数、多功能混杂函数

### 3\.2 统一初始化/销毁范式（全项目强制）

所有堆内存结构体、资源类组件，必须严格遵守 `init/deinit` 成对范式，全局统一，无例外。

```zig
// 标准范式模板
pub fn init(allocator: std.mem.Allocator, opts: Options) !*Self {
    const self = try allocator.create(Self);
    self.* = .{
        .allocator = allocator,
        // 统一初始化所有成员
    };
    return self;
}

pub fn deinit(self: *Self) void {
    // 反向顺序释放所有子资源
    self.allocator.destroy(self);
}

```

强制约束：

- 所有动态分配组件**必须注入 allocator**，禁止使用全局默认分配器

- 有 init 必须有 deinit，资源成对释放，杜绝内存泄漏

- 禁止零散分配内存，所有内存申请统一收敛到 init 构造阶段

### 3\.3 函数形态区分规则

- **创建/加载/IO/系统调用类函数**：返回`!T` 主动抛错

- **高频运行时核心逻辑**：不向上抛错，仅日志告警、降级跳过，保证主流程不崩

- 所有结构体逻辑，必须实现为成员方法 `fn xxx(self: *Self)`，减少全局函数

## 四、通用错误处理规范（克制、稳定、工程化）

摒弃过度报错、滥用错误枚举的写法，采用「编译校验\+运行降级」双层策略。

### 4\.1 错误集合定义规则

- 仅定义**真实存在、可恢复/需处理**的系统级错误

- 参数非法、内部状态异常，不定义错误枚举，使用断言校验

- 禁止为细碎场景、无效场景冗余定义错误

### 4\.2 双层校验体系

- **开发阶段**：大量 `debug.assert` 校验内部不变量、线程约束、状态合法性

- **发布阶段**：断言自动剔除，无运行时开销，保证高性能

- 所有异常必须有日志记录，禁止静默吞错

### 4\.3 错误书写规范

- 禁止多层嵌套 try，复杂错误使用`catch |err|` 显式处理

- 错误必须携带日志信息，明确错误位置、原因

## 五、内存与数据结构通用规范

### 5\.1 内存分配总原则

- **全局禁止隐式内存分配**，所有分配器显式注入、显式传递

- 支持随时替换调试分配器、泄漏检测分配器，具备可观测性

- 禁止在业务逻辑中散落裸分配代码

### 5\.2 容器选型通用策略

- 普通低吞吐场景：使用标准库`ArrayList`、`HashMap`

- 高吞吐、高频追加、长缓存、流式场景：**自研专用容器**，避免标准库碎片问题

- 结构体尽量紧凑布局，合理使用位域、打包结构体，减少内存占用

### 5\.3 并发数据通用准则（无锁优先）

- 多线程场景优先使用**状态快照拷贝**通信

- 尽量避免共享可变状态，优先消息队列、只读副本

- 能无锁解决绝不加锁，减少锁竞争与死锁风险

## 六、Comptime 编译期编程使用规范（克制通用标准）

禁止炫技式滥用编译期逻辑，仅限定三类合法场景，全项目统一。

### 6\.1 允许使用 comptime 的场景

- 编译期生成枚举、类型、配置 Schema，统一数据源，避免硬编码不一致

- 编译期计算位宽、结构体大小、偏移量，保证 ABI 稳定

- 编译期静态校验、穷尽分支检查

### 6\.2 禁止使用 comptime 的场景

- 复杂业务逻辑、分支判断、循环计算

- 可在运行时简单实现的逻辑，强行上编译期元编程

- 过度抽象、提升阅读成本的编译期语法

核心思想：**编译期只做校验、生成、稳定约束，不做业务**。

## 七、注释与文档通用规范

### 7\.1 注释原则

代码优先自解释，注释只补全**代码无法表达的意图**。

- 统一使用单行 `//`，禁止多行块注释

- 对外导出 API 必须写文档注释，内部私有函数无需冗余注释

- 复杂算法、特殊约束、兼容性逻辑必须加注释说明设计意图

- 禁止复读机式注释（描述代码显而易见的行为）

### 7\.2 Git 提交规范（通用工程标准）

统一格式：`模块: 简短清晰描述`

示例：`config: fix config merge priority bug`

## 八、测试与工程构建规范

### 8\.1 单元测试规范

- 模块与测试文件分离，`xxx.zig` 对应 `xxx_test.zig`

- 测试代码严格复用业务代码规范，不搞特殊写法

- 核心工具、数据结构、解析逻辑必须全覆盖单元测试

### 8\.2 构建脚本规范

- 全程使用原生 `build.zig`，不依赖第三方构建工具

- 构建逻辑模块化、分步拆分，禁止巨型构建函数

- 区分 debug/release\-safe/release\-fast 环境，差异化开启校验与性能策略

## 九、全局通用工程红线（绝对禁止）

所有项目强制执行，违反即合并驳回

1. 禁止使用全局默认分配器，所有内存必须显式注入

2. 禁止跨模块直接读写结构体字段

3. 禁止业务代码掺杂平台判断逻辑

4. 禁止超长函数、多职责混杂函数

5. 禁止静默吞错、无日志报错

6. 禁止滥用 comptime 做业务逻辑

7. 禁止随意导出模块内部私有实现

## 十、总结

1\. 可见性最小化，默认私有、谨慎公开；

2\. 内存全显式注入，init/deinit 成对管理；

3\. 函数短小单一，职责绝对清晰；

4\. 错误分层处理，开发断言校验、生产降级稳定；

5\. comptime 只做校验生成，不写业务；

6\. 平台逻辑隔离，核心代码完全跨平台纯净；

7\. 无锁优先、快照通信，规避并发风险；

8\. 代码自解释优先，注释只补设计意图。
