# 项目编码与工程规范

## 前言

本规范目标：统一代码风格、降低维护成本、规避内存问题、提升工程健壮性、保证长期可迭代性。

## 一、通用命名规范（全局强制）

### 1.1 基础命名大小写规则

| 类别 | 规则 | 示例 |
|------|------|------|
| 结构体 / 枚举 / 联合体 / 类型别名 | `PascalCase` | `GpuState`, `ParticlePool`, `MeteorSystem` |
| 函数 / 方法 | `camelCase` | `init()`, `allocParticle()`, `fillTextInstances()` |
| 变量 / 字段 / 模块文件 | `snake_case` | `instance_count`, `heart_ready`, `gpu_state.zig` |
| 全局常量 / 编译期配置 | `SCREAMING_SNAKE_CASE` | `MAX_INSTANCES`, `POOL_CAPACITY` |
| 私有内部辅助函数 | 无前缀，依赖 `pub` 可见性控制 | `_compact()`, `_format_uint()` |

> Zig 社区惯例是函数 `camelCase`、变量/字段 `snake_case`。不使用下划线前缀区分可见性——可见性由 `pub` 关键字控制。

### 1.2 命名语义约束

- 禁止自定义非通用缩写，仅允许行业公认缩写：`ctx`、`err`、`idx`、`buf`、`ptr`、`gpu`、`dpr`、`rng`
- 命名完整表意，杜绝模糊命名：`proc`/`tmp`/`data` 等无意义命名禁止使用
- 局部短作用域变量可极简（`i`、`j`、`idx`），长生命周期变量必须语义完整
- 布尔字段/变量使用 `is_` / `has_` / `can_` 等前缀表达真值语义：`is_alive`、`has_focus`、`can_render`

### 1.3 类型别名约定

每个模块文件顶部定义常用别名，减少视觉噪音：

```zig
const std = @import("std");
const Allocator = std.mem.Allocator;
const assert = std.debug.assert;
```

`@This()` 用于结构体自引用，避免重复类型名：

```zig
pub const ParticlePool = struct {
    const Self = @This();

    pub fn init(alloc: Allocator, capacity: usize) !Self {
        // ...
    }
};
```

### 1.4 对外 API 命名规范

- 跨模块/对外暴露接口统一格式：`模块_对象_行为`（C ABI 导出）
- Zig 内部 API：`模块.类型.方法`，通过 re-export 扁平化命名空间
- C ABI 导出接口全部统一前缀（如 `oayao_`），杜绝随意命名

```zig
// C ABI 导出 — 统一前缀
export fn oayao_spawn_heart(event_id: [*:0]const u8) void { ... }
export fn oayao_sync_hearts(active_ids: [*:0]const u8) void { ... }

// Zig 内部 API — 模块命名空间
const heart = HeartSystem.init(&pool, &rng, elapsed, cx, cy, h, fp_x, fp_y, dpr);
```

## 二、访问控制与模块可见性规范（强制最小权限）

核心原则：**默认私有，仅必要公开**，从语言层面杜绝滥用导出。

### 2.1 符号导出规则

- 所有函数、结构体、常量默认不写 `pub`，默认模块私有
- 仅跨目录、跨模块调用的实体，才允许标记 `pub`
- 内部辅助结构体、工具函数、状态字段，一律私有，禁止导出

### 2.2 结构体成员访问约束

- 禁止跨模块直接读写结构体字段
- 所有字段读写、状态修改，必须通过结构体成员方法完成
- 对外封装优先使用不透明指针，隐藏所有内部实现细节

```zig
// 正确：通过方法访问
pub fn isAlive(self: Particle) bool {
    return self.flags.alive;
}

pub fn setAlive(self: *Particle, alive: bool) void {
    self.flags.alive = alive;
}
```

### 2.3 模块组织与 Re-export 模式

父级目录使用 re-export 模式统一对外接口，调用方不感知内部目录结构：

```zig
// src/renderer.zig — 父级 re-export 文件
pub const Thread = @import("renderer/Thread.zig");
pub const State = @import("renderer/State.zig");
pub const Message = @import("renderer/message.zig").Message;

// 调用方只需 import 父级模块
const renderer = @import("renderer.zig");
const thread = try renderer.Thread.init(alloc, config);
```

目录结构约定：
- 核心类型一个文件一个结构体，文件名与主类型名一致（`ParticlePool` → `pool.zig` 或 `ParticlePool.zig`）
- 子模块目录内的入口文件命名：与目录同名（`renderer/` → `renderer.zig`）或使用模块主类型名
- 辅助工具、常量可合并到同一文件

### 2.4 平台代码隔离规则

- 所有系统、平台、OS 专属逻辑，全部收敛到独立 `platform/` 或 `os/` 层
- 业务核心代码**禁止出现任何平台判断逻辑**
- 条件编译仅允许在适配层使用，上层完全平台无关

```zig
// platform/bootstrap.zig — 平台适配层
const builtin = @import("builtin");

pub fn bootstrap() void {
    if (builtin.os.tag == .ios) {
        oayao_swift_bootstrap();
    }
}
```

```zig
// main.zig — 上层完全平台无关
const bootstrap = @import("platform/bootstrap.zig");
export fn init() void {
    // ...
    bootstrap.bootstrap();
}
```

## 三、函数设计通用规范（高可维护性核心）

### 3.1 函数尺寸与职责约束

- 单函数严格遵循**单一职责原则**
- 常规函数行数控制在 **20–60 行**
- 超过 80 行的函数必须拆分为私有子辅助函数
- 禁止巨型函数、多职责混杂函数

### 3.2 统一初始化/销毁范式（全项目强制）

所有堆内存结构体、资源类组件，必须严格遵守 `init/deinit` 成对范式，全局统一，无例外。

**标准模板：**

```zig
const Allocator = std.mem.Allocator;

pub const Options = struct {
    capacity: usize = 5000,
    enable_audio: bool = false,
};

pub const MyComponent = struct {
    alloc: Allocator,
    buf: []u8,
    child: ChildType,

    pub fn init(alloc: Allocator, opts: Options) !MyComponent {
        const buf = try alloc.alloc(u8, opts.capacity);
        errdefer alloc.free(buf);  // 逐资源 errdefer 保护

        var child = try ChildType.init(alloc);
        errdefer child.deinit();

        return .{
            .alloc = alloc,
            .buf = buf,
            .child = child,
        };
    }

    pub fn deinit(self: *MyComponent) void {
        self.child.deinit();
        self.alloc.free(self.buf);
        self.* = undefined;  // 防止 use-after-free
    }
};
```

强制约束：

- 所有动态分配组件**必须注入 allocator**，禁止使用全局默认分配器
- 有 `init` 必须有 `deinit`，资源成对释放，杜绝内存泄漏
- `init` 中每获取一个资源，立即 `errdefer` 释放，保证异常安全
- `deinit` 中设置 `self.* = undefined`，在 Debug/ReleaseSafe 模式下捕获悬垂指针访问
- 复杂初始化参数使用 `Options` 子结构体封装，提供默认值

### 3.3 函数形态区分规则

- **创建/加载/IO/系统调用类函数**：返回 `!T` 主动抛错（如 `init`, `loadFile`, `connect`）
- **高频运行时核心逻辑**：不向上抛错，仅日志告警、降级跳过，保证主流程不崩（如 `update`, `render`, `tick`）
- 所有结构体逻辑，必须实现为成员方法 `fn xxx(self: *Self)`，减少全局函数

### 3.4 defer vs errdefer 明确区分

`defer` 和 `errdefer` 语义不同，错用直接导致资源泄漏：

| 关键字 | 触发时机 | 适用场景 |
|--------|----------|----------|
| `defer` | 当前作用域退出时**总是**执行 | 非 `init` 函数中的资源清理、锁释放 |
| `errdefer` | 当前作用域**因错误退出时**才执行 | `init` 中保护已分配资源，仅失败时回滚 |

```zig
// 正确：init 中使用 errdefer — 仅在构造失败时回滚
pub fn init(alloc: Allocator) !Self {
    const a = try alloc.alloc(u8, 100);
    errdefer alloc.free(a);   // 失败 → 释放 a
    const b = try loadFile(); // 如果失败 → errdefer 释放 a
    errdefer b.deinit();      // 失败 → 释放 b
    return .{ .a = a, .b = b };
}

// 正确：非 init 中使用 defer — 总是执行
pub fn process(self: *Self) void {
    self.lock();
    defer self.unlock();      // 函数返回时总是解锁
    // ...
}

// 错误：init 中用 defer 替代 errdefer
pub fn init(alloc: Allocator) !Self {
    const a = try alloc.alloc(u8, 100);
    defer alloc.free(a);      // 即使 init 成功也会释放 a！
    return .{ .a = a };       // 返回值中 a 已被释放 → use-after-free
}
```

核心原则：**`init` 用 `errdefer`（失败才回滚），其他函数用 `defer`（总是执行）**。

### 3.5 位标志使用 packed struct

布尔标志集合使用 `packed struct` 紧凑存储：

```zig
const ParticleFlags = packed struct(u8) {
    alive: bool,
    immortal: bool,
    floating: bool,
    beat: bool,
    meteor: bool,
    fading_out: bool,
    _pad: u2 = 0,
};
```

优势：一个 `u8` 存 6 个布尔值，通过 `@bitCast` 或直接赋值操作，零开销。

### 3.6 泛型类型参数模式（零成本抽象）

当多个组件共享相同结构但依赖不同的具体类型时，使用 `fn(comptime T: type) type` 参数化：

```zig
// 泛型渲染器：Metal 和 OpenGL 共享同一套渲染逻辑
fn GenericRenderer(comptime GraphicsAPI: type) type {
    return struct {
        const Self = @This();

        api: GraphicsAPI,

        pub fn drawFrame(self: *Self) void {
            const target = GraphicsAPI.Target.init();
            const pass = self.api.createRenderPass(target);
            // ... 平台无关的渲染管线
        }
    };
}

// 编译期实例化 — 分别生成两套独立代码，各自针对具体 API 优化
const MetalRenderer = GenericRenderer(MetalAPI);
const OpenGLRenderer = GenericRenderer(OpenGLAPI);
```

适用场景：
- 多后端支持（GPU、字体、音频），共享逻辑但 API 不同
- 数据结构按元素类型参数化（如 `Stream(Handler)`）
- 避免运行时虚函数开销，编译器为每个实例独立优化

约束：
- 泛型内逻辑必须对所有类型实例通用
- 仅当至少 2 个真实实例存在时才抽象为泛型（避免过早抽象）

### 3.7 Tagged Union（枚举联合）模式

需要同一字段在不同状态下存储不同类型数据时，使用枚举标记的联合体：

```zig
// 粒子存活时存储 birth_sec，回收到自由链表时存储 next_free 索引
pub const Particle = struct {
    _storage: union {
        birth_sec: f32,    // 存活态：出生时间戳
        next_free: usize,  // 死亡态：自由链表下一节点索引
    },
    // ...
};
```

适用场景：
- 同一内存位置在不同生命周期阶段有不同含义
- 节省内存 — 两个字段共享同一块存储空间
- 相比 `?` 可选类型，union 显式枚举所有可能状态

约束：
- 必须通过上下文明确当前使用哪个变体（不能无标记读取错误变体）
- 变体字段数量不宜过多（超过 3 个考虑拆分为独立结构体）

### 3.8 switch 穷尽性驱动设计

利用 Zig 编译器对 `switch` 穷尽性检查，迫使代码覆盖所有枚举变体：

```zig
// 枚举新增 variant 时，编译器强制所有 switch 处同步更新
pub const Action = enum {
    move_cursor,
    erase_line,
    scroll_up,
    // 新增 print_unicode → 下面所有 switch 处编译器报错，杜绝遗漏
    print_unicode,
};

pub fn handle(action: Action) void {
    switch (action) {
        .move_cursor => self.doMove(),
        .erase_line => self.doErase(),
        .scroll_up => self.doScroll(),
        // 缺少 .print_unicode → 编译错误！
    }
}
```

设计原则：
- 涉及状态机、消息分发、事件处理的地方优先使用 `switch` + `enum` 而非 `if-else` 链
- 不要给 `switch` 加 `else` 分支吞掉未处理变体（除非确实有明确的默认行为）
- 新增枚举成员时，编译器报错即 To-Do 清单 — 每个错误对应一处需要适配的代码

> Ghostty 的 VT Parser 定义了 75+ 个 `Action` 枚举变体，依赖 switch 穷尽性保证所有 escape sequence 都有对应处理逻辑，从未出现遗漏 case 的运行时 bug。

## 四、通用错误处理规范（克制、稳定、工程化）

摒弃过度报错、滥用错误枚举的写法，采用「编译校验 + 运行降级」双层策略。

### 4.1 错误集合定义规则

- 仅定义**真实存在、可恢复/需处理**的系统级错误（`OutOfMemory`、`FileNotFound` 等）
- 参数非法、内部状态异常，不定义错误枚举，使用断言校验
- 禁止为细碎场景、无效场景冗余定义错误

### 4.2 双层校验体系

- **开发阶段**：大量 `std.debug.assert` 校验内部不变量、线程约束、状态合法性
- **发布阶段**（`ReleaseSafe` / `ReleaseFast`）：断言自动剔除，无运行时开销
- 所有异常必须有日志记录，禁止静默吞错

### 4.3 错误书写规范

**try 用于简单传播：**

```zig
const gpu = try GpuState.init(alloc);
const pool = try ParticlePool.init(alloc, POOL_CAPACITY);
```

**catch |err| switch 用于需要区分错误类型的场景：**

```zig
const file = try std.fs.openFileAbsolute(path, .{}) catch |err| switch (err) {
    error.FileNotFound => {
        log.warn("config file not found, using defaults path={s}", .{path});
        return defaultConfig();
    },
    else => return err,
};
```

**catch unreachable 用于可证明不会失败的调用：**

```zig
// 确定性计算不会失败
const result = buf.writer().writeAll(data) catch unreachable;
```

**comptime assert 窄化错误类型：**

```zig
fn fallible() error{ A, B, C }!void { ... }

fn caller() !void {
    fallible() catch |err| {
        comptime assert(@TypeOf(err) == error{A, B, C});
        // 现在 err 类型已被窄化，switch 必须穷尽
        switch (err) {
            error.A => handleA(),
            error.B => handleB(),
            error.C => handleC(),
        }
    };
}
```

### 4.4 日志规范

每个模块文件定义 scoped logger，日志携带模块上下文：

```zig
const log = std.log.scoped(.heart_system);
const log = std.log.scoped(.gpu);

// 使用
log.debug("heart contour spawned count={d}", .{CONTOUR_COUNT});
log.warn("allocation failed for event_id, discarding", .{});
log.err("GPU init failed: {}", .{err});
```

生产代码中关键路径保留 `log.debug`（可通过编译选项剔除），异常路径必须 `log.warn` 或 `log.err`。

## 五、内存与数据结构通用规范

### 5.1 内存分配总原则

- **全局禁止隐式内存分配**，所有分配器显式注入、显式传递
- 支持随时替换调试分配器、泄漏检测分配器，具备可观测性

```zig
// 允许：入口处统一切换调试分配器
const APP_ALLOCATOR = std.heap.c_allocator;
// const APP_ALLOCATOR = std.heap.GeneralPurposeAllocator(.{}){}.allocator(); // 调试用
```

- 禁止在业务逻辑中散落裸分配代码

### 5.2 容器选型通用策略

- 普通低吞吐场景：使用标准库 `ArrayList`、`HashMap`、`StringHashMap`
- 高吞吐、高频追加、长缓存场景：**自研专用容器**（如对象池、环形缓冲），避免标准库碎片问题
- 结构体尽量紧凑布局，合理使用 `packed struct` 位域，减少内存占用

```zig
// 对象池模式：free-list 复用，避免频繁 alloc/free
pub fn allocParticle(self: *ParticlePool, pos: Vec2, elapsed: f32, opts: ParticleOpts, rng: *Rng) *Particle {
    if (self.free_head < SENTINEL) {
        const idx = self.free_head;
        self.free_head = self.particles[idx].getNextFree();
        self.particles[idx] = Particle.init(pos, elapsed, opts, rng);
        return &self.particles[idx];
    }
    // ...
}
```

### 5.3 Arena 分配器使用场景

对象图生命周期一致（如单帧临时数据、配置加载）使用 Arena 分配器，一次释放全部子对象：

```zig
var arena = std.heap.ArenaAllocator.init(alloc);
defer arena.deinit();
const aa = arena.allocator();

// 所有临时分配走 aa，defer 一次全部释放
const temp_list = try std.ArrayList(u8).initCapacity(aa, 1024);
```

### 5.4 延迟初始化模式

昂贵资源直到首次真实需要时才创建，使用 `?` 可选类型或 `undefined` + 标志位：

```zig
// 模式 A：可选类型 — 简单，有运行时检查开销
alternate_screen: ?Screen = null,

pub fn getAltScreen(self: *Self) !*Screen {
    if (self.alternate_screen == null) {
        self.alternate_screen = try Screen.init(self.alloc, .{ .cols = 80, .rows = 24 });
    }
    return &self.alternate_screen.?;
}

// 模式 B：状态标志 — 适合高频访问路径，避免可选类型分支
heart_ready: bool = false,
heart: HeartSystem = undefined,

pub fn initSystems(self: *App, w: f32, h: f32, elapsed: f32) void {
    if (self.heart_ready) return;
    self.heart = HeartSystem.init(&self.pool, &self.rng, elapsed, w, h, ...);
    self.heart_ready = true;
}
```

适用场景：
- 备用屏幕（alt screen）、调试面板等不常使用的功能模块
- 初始化成本高但未必使用的资源（大容量缓冲区、纹理）
- 条件性功能（仅在特定配置开启时分配）

约束：
- 模式 B 的 `xxx_ready` 标志必须与关联数据放同一结构体，避免分散
- 延迟初始化的资源仍须在 `deinit` 中条件释放

### 5.5 引用计数共享不可变资源

多实例共享、读多写少的资源，使用引用计数管理生命周期，避免重复加载或过早释放：

```zig
// 字体栅格 — 多个 surface 共享相同字体配置时只加载一次
pub const SharedGrid = struct {
    ref_count: usize = 1,
    atlas: Atlas,
    glyph_cache: GlyphCache,

    pub fn retain(self: *SharedGrid) void {
        self.ref_count += 1;
    }

    pub fn release(self: *SharedGrid, alloc: Allocator) void {
        self.ref_count -= 1;
        if (self.ref_count == 0) {
            self.atlas.deinit();
            self.glyph_cache.deinit();
            alloc.destroy(self);
        }
    }
};
```

适用场景：
- 字体纹理图集、着色器程序、主题配色等全局只读配置
- 多窗口/多 surface 共享的 GPU 资源

约束：
- 共享资源必须真正不可变（创建后不再修改），否则需要 `RwLock` 保护
- 引用计数操作本身不是线程安全的 — 跨线程共享时需要原子操作或外置锁

### 5.6 哨兵值与边界标记

使用特殊值标记链表结尾、空闲槽位或无效状态，避免额外的布尔字段：

```zig
// 自由链表哨兵 — maxInt(usize) 表示链表尾
const SENTINEL: usize = std.math.maxInt(usize);

pub fn allocParticle(self: *ParticlePool, ...) *Particle {
    if (self.free_head < SENTINEL) {
        // 从自由链表取节点复用
        const idx = self.free_head;
        self.free_head = self.particles[idx].getNextFree();
        return &self.particles[idx];
    }
    // 自由链表为空，分配新节点
    // ...
}
```

```zig
// C 枚举哨兵 — 强制 int 大小和穷尽性
pub const Action = enum(c_int) {
    move_cursor,
    erase_line,
    _MAX_VALUE = GHOSTTY_ENUM_MAX_VALUE, // C 兼容性哨兵，不计入业务逻辑
};
```

适用场景：
- 自由链表空节点标记
- C ABI 兼容枚举的强制大小标记
- 数组越界哨兵（类似 C 的 null terminator）

### 5.7 并发数据通用准则（无锁优先）

- 多线程场景优先使用**状态快照拷贝**通信
- 尽量避免共享可变状态，优先消息队列、只读副本
- 能无锁解决绝不加锁，减少锁竞争与死锁风险

## 六、Comptime 编译期编程使用规范（克制通用标准）

禁止炫技式滥用编译期逻辑，仅限定四类合法场景，全项目统一。

### 6.1 允许使用 comptime 的场景

1. 编译期生成枚举、类型、配置 Schema，统一数据源，避免硬编码不一致
2. 编译期计算位宽、结构体大小、偏移量，保证 ABI 稳定
3. 编译期静态校验、穷尽分支检查
4. **编译期平台选择**：GPU 后端、OS 路径等使用 `comptime` + `builtin` 静态分发

```zig
// 编译期 GPU 后端选择 — 零运行时开销
const gpu_backend = if (builtin.os.tag == .macos) .metal else .opengl;
const Renderer = switch (build_config.renderer) {
    .metal => GenericRenderer(Metal),
    .opengl => GenericRenderer(OpenGL),
};
```

```zig
// 编译期穷尽校验
comptime assert(@TypeOf(err) == error{OutOfMemory}); // 确保错误类型已窄化
```

### 6.2 禁止使用 comptime 的场景

- 复杂业务逻辑、分支判断、循环计算
- 可在运行时简单实现的逻辑，强行上编译期元编程
- 过度抽象、提升阅读成本的编译期语法

核心思想：**编译期只做校验、生成、稳定约束、平台选择，不做业务**。

## 七、注释与文档通用规范

### 7.1 注释原则

代码优先自解释，注释只补全**代码无法表达的意图**。

- 统一使用单行 `//`，禁止多行块注释
- 对外导出类型使用 `///` 文档注释：`/// A particle pool with free-list allocation.`
- 模块文件头部使用 `//!` 模块级注释：`//! GPU rendering state and pipeline management.`
- 对外导出 API 必须写文档注释，内部私有函数无需冗余注释
- 复杂算法、特殊约束、兼容性逻辑必须加注释说明设计意图
- 禁止复读机式注释（描述代码显而易见的行为）

### 7.2 注释示例

```zig
//! Particle animation and rendering system.
//!
//! Particles are managed through a free-list pool for zero-allocation reuse.
//! Each particle has a configurable lifespan, velocity, and rendering flags.

/// A pooled particle instance with position, velocity, and lifecycle state.
/// Uses packed flags for memory efficiency and a tagged union for storage reuse.
pub const Particle = struct {
    pos: Vec2,
    vel: Vec2,
    // ...
};

// Immortal particles skip the normal lifespan check — used for heart contour
// points that must persist until explicitly removed.
if (!self.flags.immortal and self.lifespan < 0.0) {
    self.flags.alive = false;
}
```

### 7.3 Git 提交规范（通用工程标准）

统一格式：`模块: 简短清晰描述`

示例：
- `heart: fix floating particle lifetime calculation`
- `gpu: add instanced rendering for text glyphs`
- `build: add iOS simulator target support`

## 八、测试与工程构建规范

### 8.1 单元测试规范

**测试组织方式：** 优先使用 Zig 的 inline test block，测试与源代码共置：

```zig
// src/particles/pool.zig — 源代码底部
test "allocParticle reuses freed slot" {
    const testing = std.testing;
    const alloc = testing.allocator;
    var pool = try ParticlePool.init(alloc, 100);
    defer pool.deinit();

    var rng = Rng.init(12345);
    const p0 = pool.allocParticle(Vec2{ .x = 0, .y = 0 }, 0.0, .{}, &rng);
    _ = pool.allocParticle(Vec2{ .x = 1, .y = 1 }, 0.0, .{}, &rng);
    p0.setAlive(false);
    pool.collectAlive();

    const p2 = pool.allocParticle(Vec2{ .x = 2, .y = 2 }, 0.0, .{}, &rng);
    try testing.expectEqual(p0, p2);  // 复用已释放的槽位
}
```

测试要求：
- 测试描述命名格式：`"被测行为"` 或 `"主题: 具体场景"`（如 `"cursor: default uses configured style"`）
- 使用 `const testing = std.testing; const alloc = testing.allocator;` 标准模板
- `testing.allocator` 会自动检测内存泄漏，无需手动泄漏检查
- 测试代码严格复用业务代码规范，不搞特殊写法
- 核心工具、数据结构、解析逻辑必须全覆盖单元测试
- 单独的测试入口文件（如 `tests.zig`）用于聚合所有 inline test

```zig
// src/tests.zig — 聚合入口
test {

}
```

### 8.2 构建脚本规范

- 全程使用原生 `build.zig`，不依赖第三方构建工具
- 构建逻辑模块化拆分：大型项目将构建逻辑拆分为独立模块（如 `src/build/` 目录下的多个文件），每个模块对应一个构建产物
- 区分 `Debug` / `ReleaseSafe` / `ReleaseFast` 环境，差异化开启校验与性能策略

```zig
// build.zig — 模块化构建示例
const app_mod = b.createModule(.{
    .root_source_file = b.path("src/main.zig"),
    .target = target,
    .optimize = optimize,
    .imports = &.{
        .{ .name = "sokol", .module = mod_sokol },
    },
});

// 不同平台不同产物
if (is_web) {
    const link_step = try sokol_build.emLinkStep(b, .{ ... });
    const web_step = b.step("web", "Build for web");
    web_step.dependOn(&link_step.step);
} else if (target.result.os.tag == .ios) {
    const app_step = try createIosAppBundle(b, lib, lib_sokol, target);
    const install_app = b.step("ios-app", "Build iOS .app bundle");
    install_app.dependOn(app_step);
} else {
    const exe = b.addExecutable(.{ ... });
    const run_cmd = b.addRunArtifact(exe);
    const run_step = b.step("run", "Run on desktop");
    run_step.dependOn(&run_cmd.step);
}
```

构建系统原则：
- 每个 build step 对应一个明确命名的构建命令（如 `zig build web`、`zig build ios-app`）
- 第三方依赖通过 `build.zig.zon` 声明版本和哈希，不手动管理源码
- 平台相关的构建逻辑使用 `if (target.result.os.tag == .xxx)` 分支，保持 build 文件可读

### 8.3 build.zig.zon 依赖管理

所有第三方依赖通过 `build.zig.zon` 锁定版本和完整性哈希：

```zig
.{
```

## 九、并发与线程模型规范

### 9.1 线程架构原则

- **线程按职责分离**：IO 线程、渲染线程、主线程各司其职
- 每个线程持有明确的资源集合，不跨线程共享裸指针
- 线程间通过**消息队列**或**异步通知**通信

### 9.2 消息传递模式

优先使用固定大小消息队列传递数据，避免共享可变状态：

```
[IO Thread]  --message--> [Renderer Thread]
      |                          |
  写入 pty                    读取终端状态
  解析 VT                     构建 GPU Buffer
  修改 Terminal               绘制帧
```

### 9.3 无锁快照模式

渲染线程需要读取终端状态时，在锁保护下快速拷贝关键数据，然后释放锁进行耗时操作：

```zig
// 伪代码：快照模式
fn renderFrame(self: *Renderer) void {
    // 1. 持锁时间尽可能短 — 仅拷贝
    self.state.mutex.lock();
    defer self.state.mutex.unlock();
    const snapshot = self.state.terminal.snapshot();
    // 2. 释放锁后基于快照渲染，不阻塞 IO 线程
    self.buildGpuBuffers(snapshot);
}
```

### 9.4 全局状态管理

- 尽量避免全局可变状态
- 必须的全局状态（如 sokol C ABI 回调所需的 app 指针）集中声明，注释说明必要性：

```zig
// Global app pointer — the minimum necessary global for sokol's C ABI
// callbacks which provide no userdata parameter.
var g_app: ?*App = null;
```

- 所有对外部暴露的 C ABI 回调通过该全局指针访问，不扩散全局变量模式

### 9.5 脏标记与假阳性优化

渲染/同步场景中，使用「允许假阳性、绝不假阴性」的脏标记策略，减少判定开销：

```
概念：脏标记表示"可能发生了变化"，不是"一定发生了变化"。
      检查发现标记置位时，做全量检查；标记未置位时，保证无变化。
```

```zig
// Row 的脏标记 — 假阳性允许，假阴性禁止
pub const Row = packed struct(u64) {
    dirty: bool,               // 真阳性或假阳性（允许）
    has_grapheme: bool,        // 真阳性或假阳性（允许）
    has_hyperlink: bool,       // 真阳性或假阳性（允许）
    // 关键：这些标记的 false 值保证为真阴性。
    // has_grapheme == false → 该行一定没有任何 grapheme。
    // 因此 VT 操作 (erase/delete) 可安全跳过 grapheme 拆解。
};
```

这个模式的价值：VT 的抹行操作（`erase_line`）本需逐单元拆解 grapheme 集群，但绝大多数行从未写入过 grapheme。`has_grapheme` 标记让 `erase_line` 在读标记值为 `false` 时直接跳过拆解逻辑，实测 4x 性能提升。标记偶尔为 `true`（假阳性）时多做一次拆解无副作用；标记为 `false`（真阴性）时跳过的是确定不需要的工作。

适用场景：
- 渲染帧间状态同步（dirty rect / dirty region）
- 终端网格行级装饰属性标记（grapheme、hyperlink、semantic prompt）
- 缓存失效判定（可能过期 vs 确定新鲜）
- 事件合并/去重窗口

设计原则：
- **伪代码**：`if maybeDirty { doFullCheck(); }` — 假阳性的代价是一次多余检查，假阴性的代价是数据不一致
- 脏标记的清除时机必须与数据更新同一临界区
- 计数型标记（版本号/代数）可替代布尔标记，解决 ABA 问题

## 十、全局通用工程红线（绝对禁止）

所有项目强制执行，违反即合并驳回：

1. 禁止使用全局默认分配器，所有内存必须显式注入
2. 禁止跨模块直接读写结构体字段
3. 禁止业务代码掺杂平台判断逻辑
4. 禁止超长函数（>80 行）、多职责混杂函数
5. 禁止静默吞错、无日志报错
6. 禁止滥用 comptime 做业务逻辑
7. 禁止随意导出模块内部私有实现
8. 禁止全局可变状态的扩散（仅允许最小必要全局，须注释说明）
9. 禁止 `defer` 替代 `errdefer` 放在 `init` 的资源保护中
10. 禁止 `switch` 中使用 `else` 分支吞掉未处理枚举变体（除非有明确默认行为）
11. 禁止延迟初始化的资源在 `deinit` 中遗漏条件释放
12. 禁止依赖版本使用 branch/tag 浮动引用，必须锁定 commit hash

## 十一、总结

1. 函数 `camelCase`，变量/字段 `snake_case`，类型 `PascalCase`，常量 `SCREAMING_SNAKE_CASE`
2. 可见性最小化，默认私有、谨慎公开；用 re-export 模式组织模块
3. 内存全显式注入，`init` 用 `errdefer` 保护资源，`deinit` 设置 `self.* = undefined`
4. `init` 用 `errdefer`（失败回滚），其他函数用 `defer`（总是执行）
5. 函数短小单一，职责绝对清晰
6. 错误分层处理：`try` 简单传播，`catch |err| switch` 区分处理，`catch unreachable` 确定性忽略
7. 每个文件定义 `const log = std.log.scoped(.module);`，异常必日志
8. `comptime` 只做校验、生成、平台选择、类型参数化，不写业务
9. 平台逻辑隔离在 `platform/` 层，核心代码完全跨平台纯净
10. 无锁优先、快照通信、消息队列，规避并发风险
11. 脏标记允许假阳性、绝不允许假阴性 — `if maybeDirty { doFullCheck(); }`
12. 代码自解释优先，注释只补设计意图；`///` 文档注释，`//!` 模块注释
13. 内联测试与源码共置，`testing.allocator` 自动泄漏检测
14. `packed struct` 紧凑位标志，`Options` 子结构体封装复杂参数
15. 泛型 `fn(comptime T: type) type` 做零成本抽象，tagged union 做变体数据
16. `switch` 穷尽性驱动设计 — 枚举新增成员时编译器报错即 To-Do 清单
17. 构建逻辑模块化拆分，依赖锁定 commit hash，代码生成提交版本管理
